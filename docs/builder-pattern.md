## Builder pattern quick start

`JavaBeanTester.builder(...)` is the supported entry point for this library.

The typical flow is:

1. Start with `JavaBeanTester.builder(YourType.class)`
2. Enable any optional checks you need
3. Finish with a terminal method such as `test()`, `testInstance(...)`, `testEquals(...)`,
   `testObjectMethods()`, or `testPrivateConstructor()`

## Common usage patterns

### Smoke test a bean

```java
JavaBeanTester.builder(YourBean.class).test();
```

This runs getter/setter checks, exercises public constructors, runs `clear()` when present, and
performs serialization round-trip checks for types that implement `Serializable` or
`Externalizable`.

### Include equals/hashCode/toString checks

```java
JavaBeanTester.builder(YourBean.class).checkEquals().loadData().test();
```

Use `checkEquals()` when the bean implements equality semantics and you want the tester to verify
`equals`, `hashCode`, `toString`, and `canEqual`.

### Compare two existing instances

```java
JavaBeanTester.builder(YourBean.class).loadData().testEquals(left, right);
```

This is useful when the instances are created by your test and only the equality path needs to be
verified.

### Populate a specific instance

```java
JavaBeanTester.builder(YourBean.class).loadData().testInstance(bean);
```

Use `loadData()` when the test should fill nested values before validating getters and setters.

### Skip selected properties

```java
JavaBeanTester.builder(YourBean.class).checkEquals().loadData().skip("id", "timestamp").test();
```

`skip(...)` excludes named bean properties from the getter/setter pass.

### Private constructor coverage

```java
JavaBeanTester.builder(UtilityClass.class).testPrivateConstructor();
```

### Object-method-only coverage

```java
JavaBeanTester.builder(YourBean.class).testObjectMethods();
```

## When to use the second builder overload

Prefer:

```java
JavaBeanTester.builder(YourBean.class)
```

The single-argument builder automatically chooses the best extension strategy:

* final classes fall back to an `Object` comparison type
* Lombok builder types skip generated extensions automatically
* other classes use a generated Byte Buddy extension for equality-related checks

Use the two-argument overload when you need to control that behavior:

```java
JavaBeanTester.builder(YourBean.class, YourExtension.class)
```

or explicitly disable extension generation:

```java
JavaBeanTester.builder(YourBean.class, null)
```

Passing `null` is useful for types that cannot or should not be subclassed during a test.

## Migration notes

If you are upgrading older tests, translate them to the builder in two steps:

1. Move the class under test into `JavaBeanTester.builder(...)`
2. Replace direct one-shot helper usage with builder options plus a terminal method

Examples:

* bean-wide validation -> `JavaBeanTester.builder(YourBean.class).test()`
* bean-wide validation with object-method checks ->
  `JavaBeanTester.builder(YourBean.class).checkEquals().test()`
* equality-only checks on prepared instances ->
  `JavaBeanTester.builder(YourBean.class).testEquals(left, right)`
* private-constructor checks ->
  `JavaBeanTester.builder(UtilityClass.class).testPrivateConstructor()`
